# North Star

## The Ultimate Direction

O.M.A.-C.O.R.E. has one ultimate success metric:

**The user consistently makes better decisions than they would have made without O.M.A.-C.O.R.E.**

This metric is not about the system's performance. It is about the user's performance, augmented by the system. If the user's decisions do not improve, the system has failed — regardless of its internal metrics, trade profitability, or architectural elegance.

## Why This Metric

Profit alone is insufficient for three reasons:

**First, profit can come from luck.** A system that profits for a year through random chance is not necessarily making good decisions. The metric should measure decision quality, not outcome luck.

**Second, profit is domain-specific.** A trading system is evaluated by trading profit. But O.M.A.-C.O.R.E. is not only a trading system. Its value extends to creation, entrepreneurship, and any domain where criterion improves decisions. The ultimate metric must be domain-independent.

**Third, profit measures the past.** Criterion should improve the future. A system that profited yesterday but learned nothing is less valuable than a system that broke even yesterday but learned something that improves tomorrow's decisions.

The user's decision quality — measured across all domains where the system operates — is the only metric that captures all three concerns.

## What Criterion Should Improve

Criterion should eventually improve the user's access to scarce resources:

**Capital.** Financial resources. The most tangible and measurable scarce resource. Trading is the initial validation domain because capital is the easiest resource to measure.

**Time.** The scarcest resource. Criterion should help the user spend less time on noise and more time on what matters. A system that generates more decisions without improving their quality is consuming time, not saving it.

**Attention.** The ability to focus on what matters. Criterion should help the user direct attention toward emerging consequences and away from mature or illusory ones.

**Knowledge.** The user's own understanding should deepen through interaction with the system. The system should not only make decisions — it should explain them, justify them, and teach from them.

**Relationships.** In the entrepreneur and creator domains, criterion should help the user identify which relationships matter, which collaborations have the highest potential, and which connections are worth investing time in.

**Mobility.** Freedom to move — between markets, between domains, between strategies. Criterion should make the user less dependent on any single approach and more able to adapt to changing conditions.

**Health.** The ultimate scarce resource. No amount of criterion matters if the user's health fails. The system should not optimize for profit at the expense of the user's well-being.

**Freedom of decision.** The ability to choose what to work on, when to work on it, and whether to act at all. The system should increase the user's options, not constrain them.

## The Filter

Every future feature, every proposed capability, every architectural change should be evaluated against one question:

**Does this improve the user's ability to make better decisions?**

If the answer is unclear, the proposal requires more evidence before it can be justified.

If the answer is no, the proposal should be rejected regardless of how elegant or innovative it appears.

If the answer is yes, the proposal should specify how the improvement will be measured and validated.

## The Long View

The North Star does not change with technology shifts, market conditions, or architectural fashion.

A better AI model does not change it. A new programming language does not change it. A market crash does not change it. A decade of development does not change it.

The North Star is:

**Better decisions over time.**

Everything else is subordinate.
