# MVP Redefinition

## The First Operational Criterion Laboratory

*Version 1.0 — June 2026*

---

## 0. Preamble

ERA I established the scientific foundation. Seventeen first-principles documents define what criterion is, how it develops, how it must be validated, and what the project commits to reject.

ERA II begins the scientific demonstration. The question is no longer "what should we build?" but "does what we built produce evidence of criterion development?"

The original MVP was a trading prototype. It validated the event-to-action pipeline. It proved the system could detect events, generate signals, execute trades, and record outcomes reliably. This was the correct objective for ERA I.

ERA I is complete. The MVP must evolve.

This document redefines the MVP as the first operational criterion laboratory. Its purpose is no longer producing trades. Its purpose is producing evidence.

---

## PART I: Why the MVP Must Change

### 1.1 The Original MVP Was Designed for a Different Question

The original MVP asked: *"Can we build a system that detects events, generates signals, and executes trades safely?"*

That question has been answered. The MVP demonstrates reliable event detection, signal generation through agent voting, safe execution through guard layers, and outcome recording with agent-level attribution. The event-to-action pipeline works.

But ERA I revealed that the event-to-action pipeline is only the prerequisite. The real question — the question that justifies the project's existence — is: *"Can the system develop criterion over time?"*

The original MVP cannot answer this question. It has no hypotheses. It has no evidence tracking. It has no error classification by decision type. It has no knowledge extraction. It has no criterion measurement. It is a functioning signal engine that cannot learn from its own outcomes.

### 1.2 Trading Is the First Experimental Domain, Not the Identity

The project uses trading as its first validation domain because trading provides fast feedback, unambiguous outcomes, and abundant data. Trading is a favorable environment for developing and testing the criterion framework.

But trading is not the project's identity. The project is not a trading system. It is a scientific research program that happens to use trading as its first laboratory.

This distinction is critical. If the project were a trading system, its success would be measured by profit. If it is a scientific research program using trading as a laboratory, its success is measured by evidence of criterion development — and profit is a consequence, not the goal.

### 1.3 The MVP Must Become a Criterion Laboratory

A laboratory is not defined by what it produces. It is defined by the quality of experiments it enables.

The original MVP produces trades. The redefined MVP must produce experiments, evidence, and knowledge.

The transition:

| Dimension | Trading Prototype | Criterion Laboratory |
|-----------|-------------------|---------------------|
| Primary output | Trades, signals, PnL | Evidence, knowledge, criterion metrics |
| Unit of work | Trade cycle | Hypothesis lifecycle |
| Success metric | Profit factor, win rate | Criterion score dimensions |
| Feedback signal | PnL per trade | Outcome vs. hypothesis comparison |
| Learning mechanism | Council weight adjustment | Full hypothesis lifecycle (creation → evidence → decision → outcome → reflection → knowledge → criterion update) |
| Memory | Trade records | Scientific record + operational memory + semantic memory |
| Role of failure | Loss to be avoided | Evidence to be analyzed |
| Time horizon | Per-trade or per-session | Per-hypothesis lifecycle (hours to years) |

### 1.4 Why This Change Is Necessary Now

The philosophy is complete. The scientific method is designed. The research protocol is written. But none of these produce evidence on their own. Evidence comes from the MVP operating within a framework that captures, analyzes, and learns from every cycle.

Delaying this change means accumulating more operational data without learning. Every day the MVP runs without a hypothesis lifecycle is a day of evidence not captured, knowledge not extracted, and criterion not measured.

## PART II: The Purpose of the MVP

### 2.1 The New Mission

The MVP exists to answer one scientific question:

> *Did the system improve its criterion during this operational cycle?*

Every cycle — every trade, every decision, every hypothesis test — produces evidence that either supports or weakens the claim that the system is developing criterion.

Everything else is secondary.

- Profit is secondary. It is evidence, not purpose.
- Win rate is secondary. It is a metric, not a mission.
- Automation is secondary. It is a capability, not an objective.
- Scalability is secondary. It is a future concern, not a present requirement.

### 2.2 What the MVP Must Demonstrate

Before the project can claim it develops criterion, the MVP must demonstrate:

1. **Hypotheses are formed.** The system generates testable, falsifiable beliefs about consequences before any action is taken.

2. **Evidence is tracked.** Each hypothesis accumulates supporting and contradicting evidence from independent sources.

3. **Decisions are recorded with reasoning.** Every decision is linked to the hypothesis that produced it, the evidence available at decision time, and the confidence at decision time.

4. **Outcomes are compared to hypotheses.** After each outcome, the system evaluates whether the hypothesis was confirmed, rejected, or inconclusive.

5. **Errors are classified by type.** When a hypothesis is rejected, the error is decomposed into its components (wrong perception, wrong judgment, wrong timing, poor execution, external shock).

6. **Knowledge is extracted.** Every evaluated hypothesis produces at least one generalizable lesson.

7. **Criterion metrics are tracked.** Calibration, hypothesis quality, error recurrence, and other criterion dimensions are measured over time.

8. **Improvement is demonstrated.** After sufficient cycles, at least one criterion dimension shows a statistically significant positive trend.

### 2.3 What the MVP Is Not

The MVP is not:
- A production trading system
- A profit-optimization engine
- A fully autonomous decision-maker
- A finished product
- A demonstration of AI capability

The MVP is:
- A scientific instrument
- A hypothesis-testing apparatus
- An evidence-generation platform
- A learning system that demonstrates its learning
- The first iteration of a ten-year research program

## PART III: The Operational Cycle

### 3.1 The Complete Cycle

Every operational cycle in the redefined MVP follows this conceptual path:

```
Reality
  │
  ▼
Need ───────────────────────── A gap in criterion; something the system
  │                             cannot judge well enough
  ▼
Question ────────────────────── What must we know to act?
  │
  ▼
Competing Hypotheses ────────── Multiple possible explanations for
  │                             what is happening or will happen
  ▼
Evidence Collection ─────────── Supporting and contradicting evidence
  │                             for each hypothesis
  ▼
Hypothesis Competition ──────── Which hypothesis best explains the evidence?
  │
  ▼
Chosen Hypothesis ───────────── The hypothesis that wins the competition
  │
  ▼
Confidence ──────────────────── How likely is this hypothesis to be correct?
  │
  ▼
Conviction ──────────────────── How strongly should we act on this belief?
  │
  ▼
Decision ────────────────────── What will we do?
  │
  ▼
Execution ───────────────────── The action taken
  │
  ▼
Outcome ─────────────────────── What actually happened
  │
  ▼
Reflection ──────────────────── Compare outcome to hypothesis.
  │                             Was the hypothesis correct?
  │                             Was the decision good given the evidence?
  │                             What type of error occurred?
  ▼
Knowledge Extraction ────────── What generalizable lesson can we learn?
  │
  ▼
Criterion Update ────────────── How does this change the system's judgment?
  │
  ▼
Improved Future Decisions ───── The cycle spirals upward
```

### 3.2 Stage Descriptions

**Reality.** The world as it is. Infinite in complexity. The system observes a tiny fraction of it through its sensors and data feeds.

**Need.** A gap in the system's criterion is identified. The system cannot judge something it needs to judge. A pattern is not recognized. A consequence is not detected. An error of a known type occurs again. This need is the origin of every research cycle. Without a need, the system is not learning — it is operating.

**Question.** The need is translated into a specific, answerable question. Example: "Is this combination of events producing a gold rally, or is this noise?" The quality of the question determines the quality of everything that follows.

**Competing Hypotheses.** Multiple plausible answers to the question are generated. The system never holds a single hypothesis. It maintains at least 2–3 competing explanations. Each hypothesis specifies: what consequence is expected, under what conditions, within what time horizon, and what would disprove it.

**Evidence Collection.** Evidence is gathered for and against each hypothesis. Evidence is not averaged — supporting and contradicting evidence are tracked as separate dimensions. Source reliability, recency, and independence are recorded.

**Hypothesis Competition.** Evidence is evaluated against each competing hypothesis. The hypothesis that best explains all available evidence — not the one with the most supporting evidence — wins the competition. The spread across competing hypotheses is recorded as uncertainty.

**Chosen Hypothesis.** The winning hypothesis is selected as the basis for the decision. The losing hypotheses are not discarded — they are preserved as active competitors that may regain advantage if new evidence shifts the balance.

**Confidence.** A calibrated estimate of how likely the chosen hypothesis is to be correct. Confidence is derived from: the evidence ratio, the quality of evidence, the system's calibration history for similar hypotheses, and the uncertainty indicated by competition spread.

**Conviction.** How strongly the system acts on its confidence. Conviction translates confidence into resource allocation: position size, stop distance, attention allocation. A well-calibrated system allocates more resources to higher-confidence hypotheses.

**Decision.** The specific choice: act, wait, monitor, ignore, reduce, replace. The decision is recorded with its reasoning — which alternatives were considered, why this one was chosen, what would change the decision.

**Execution.** The action is carried out. In trading: a trade is opened or closed. Execution details (timing, price, size) are recorded alongside the decision.

**Outcome.** What actually happened. The outcome is recorded objectively before any interpretation. Raw outcome data (price movement, PnL, duration) is separated from interpreted evidence (was the hypothesis confirmed?).

**Reflection.** The most critical stage. The system compares the outcome to the hypothesis and the decision. Questions asked:
- Was the hypothesis confirmed, rejected, or inconclusive?
- Was the decision good given the evidence available at decision time?
- What type of error occurred? (wrong hypothesis, wrong timing, poor execution, external shock, or correct)
- What would the system do differently with the same evidence?
- What confounds or uncertainties remain?

**Knowledge Extraction.** A generalizable lesson is formulated. The lesson is not "this trade lost money." The lesson is: "in this context, this type of hypothesis fails because of this specific factor." Knowledge is always provisional and is recorded as a testable claim.

**Criterion Update.** Knowledge is integrated into the system's accumulated judgment. This may adjust: confidence calibration, evidence source reliability, priors for similar hypotheses, error type frequencies, or pattern recognition thresholds.

**Improved Future Decisions.** The cycle does not return to its starting point. The next cycle begins from a higher baseline — the system's criterion has been updated, its knowledge base has grown, and its next question will be better informed.

### 3.3 Why the Cycle Must Include All Stages

Every stage serves a specific function. Skipping a stage breaks the learning mechanism:

- **Skip Need:** The system generates hypotheses for every event, overwhelming itself with irrelevant questions.
- **Skip Competing Hypotheses:** The system holds one hypothesis and interprets all evidence in its favor. Confirmation bias is built in.
- **Skip Hypothesis Competition:** The system collects evidence but never evaluates which hypothesis it supports. Evidence accumulates without informing decisions.
- **Skip Reflection:** The system acts and observes but does not learn. Outcomes become data points, not knowledge.
- **Skip Criterion Update:** The system learns but does not change. Knowledge is extracted but not applied. The next cycle starts from the same baseline.

## PART IV: The Scientific Record

### 4.1 Every Cycle Generates a Permanent Record

Every operational cycle produces a scientific record. This record is the memory of the laboratory. It is immutable, temporal, queryable, and complete.

### 4.2 Record Schema

| Field | Purpose | When Recorded |
|-------|---------|---------------|
| **Decision ID** | Unique identifier for this cycle | Cycle start |
| **Research Proposal ID** | Link to the research proposal this cycle belongs to (if applicable) | Cycle start |
| **Experiment ID** | Link to the specific experiment this cycle is testing | Cycle start |
| **Timestamp** | When the cycle began | Cycle start |
| **Need** | The gap in criterion that triggered this cycle | Before hypothesis formation |
| **Question** | The specific question being asked | Before hypothesis formation |
| **Hypotheses** | All competing hypotheses, each with: consequence prediction, conditions, time horizon, invalidation conditions | Before evidence collection |
| **Rejected Hypotheses** | Which hypotheses were rejected during competition, and why | At decision time |
| **Evidence Used** | Evidence that informed the decision, with source reliability, recency, and independence for each item | At decision time |
| **Evidence Ignored** | Evidence that was available but explicitly not used, with reasoning | At decision time |
| **Chosen Hypothesis** | The hypothesis selected as the basis for action | At decision time |
| **Confidence** | Calibrated confidence in the chosen hypothesis (0.0–1.0, with uncertainty bounds) | At decision time |
| **Conviction** | How strongly the system acted on its confidence (e.g., position size as % of maximum) | At execution time |
| **Expected Scenario** | What the system expected to happen if the hypothesis is correct, including specific predictions | At decision time |
| **Contradictory Scenario** | What would contradict the hypothesis | At decision time |
| **Decision** | The specific choice made (act, wait, monitor, ignore, reduce, replace) | At decision time |
| **Decision Rationale** | Why this decision was chosen over alternatives | At decision time |
| **Action** | What was executed (trade details, content published, etc.) | At execution time |
| **Outcome** | What actually happened (raw data, before interpretation) | After outcome |
| **Outcome vs. Prediction** | Did the outcome confirm, reject, or partially align with the hypothesis? | After reflection |
| **Unexpected Events** | External factors that affected the outcome but were not part of any hypothesis | After reflection |
| **Error Classification** | If rejected, the error type(s): perception, judgment, timing, execution, external | After reflection |
| **Reflection** | Full reflection: what was learned, what was surprising, what remains unknown | After reflection |
| **Knowledge Extracted** | One or more generalizable lessons from this cycle | After reflection |
| **Criterion Change** | How the system's judgment was updated as a result of this cycle | After knowledge extraction |
| **Calibration Update** | Specific calibration adjustments made | After knowledge extraction |
| **Source Reliability Updates** | Changes to evidence source reliability scores | After knowledge extraction |
| **Future Recommendation** | What the system would do differently next time, or what it recommends investigating next | After reflection |

### 4.3 Why This Record Is the Memory of the Laboratory

The scientific record serves multiple functions that no other data structure can replace:

**It enables longitudinal analysis.** With every cycle recorded in a consistent schema, the project can analyze trends across hundreds or thousands of cycles. Did hypothesis quality improve? Did calibration converge? Did error frequencies decline?

**It prevents hindsight bias.** The record captures what was known, believed, and decided at each point in time. Future analysis cannot rewrite the past — the immutable record preserves the original reasoning.

**It enables reproduction.** Every cycle is documented in sufficient detail that it could theoretically be replayed or simulated. This is essential for validating claims of improvement.

**It enables attribution.** Every outcome can be traced back to the hypothesis, evidence, and decision that produced it. The system can answer not just "what happened?" but "why did it happen?"

**It enables meta-learning.** By analyzing the full set of scientific records, the system can learn about its own learning — which types of hypotheses succeed, which types of evidence are most informative, which error types are most common, which contexts are most challenging.

### 4.4 Storage Requirements

The scientific record must be:
- **Immutable.** Once written, it cannot be modified. Corrections are new records, not edits.
- **Temporal.** Every entry carries a timestamp. The full timeline of the system's reasoning is preserved.
- **Queryable.** Records must be searchable by any field: hypothesis type, error type, confidence range, outcome, knowledge extracted, etc.
- **Complete.** Every operational cycle produces a record. Selective recording would bias all derived analyses.
- **Linked.** Records reference each other — decisions link to hypotheses, outcomes link to decisions, knowledge links to outcomes. The graph of relationships is as important as the records themselves.

## PART V: Research Inside the MVP

### 5.1 Three Conceptual Layers

The MVP operates at three integrated layers. Each layer serves a distinct function in the research process.

```
                    ┌─────────────────────────────────────┐
                    │          LAYER 1                     │
                    │      RESEARCH PROPOSAL               │
                    │                                     │
                    │  Defines what is being investigated. │
                    │  States the research question,       │
                    │  the hypothesis, the expected        │
                    │  impact, and the success criteria.   │
                    └────────────────┬────────────────────┘
                                     │ generates
                    ┌────────────────▼────────────────────┐
                    │          LAYER 2                     │
                    │         EXPERIMENT                   │
                    │                                     │
                    │  Defines how the hypothesis will be  │
                    │  tested. Specifies the design,       │
                    │  the data, the controls, the         │
                    │  sample size, the metrics, and the   │
                    │  analysis method.                    │
                    └────────────────┬────────────────────┘
                                     │ executes
                    ┌────────────────▼────────────────────┐
                    │          LAYER 3                     │
                    │    OPERATIONAL DECISIONS             │
                    │                                     │
                    │  Every trade, every action, every    │
                    │  decision produces evidence for or   │
                    │  against the hypothesis.             │
                    └────────────────┬────────────────────┘
                                     │ produces
                    ┌────────────────▼────────────────────┐
                    │        EXPERIMENT RESULTS            │
                    └────────────────┬────────────────────┘
                                     │ extract
                    ┌────────────────▼────────────────────┐
                    │            KNOWLEDGE                 │
                    └────────────────┬────────────────────┘
                                     │ accumulates
                    ┌────────────────▼────────────────────┐
                    │           CRITERION                  │
                    └─────────────────────────────────────┘
```

### 5.2 Layer 1 — Research Proposal

A research proposal defines what is being investigated and why. It is the scientific justification for a set of experiments.

**Components:**
- **Research Question:** What specific gap in criterion does this proposal address?
- **Hypothesis:** What does the project expect to find?
- **Prior Evidence:** What is already known about this question?
- **Expected Impact:** If confirmed, how would this improve the system's criterion?
- **Success Criteria:** What would count as evidence that the hypothesis is correct?
- **Failure Criteria:** What would falsify the hypothesis?
- **Duration:** How long should this research run before evaluation?
- **Minimum Sample:** How many cycles are needed for statistical significance?

A research proposal may generate many experiments. For example, a proposal investigating "does explicit hypothesis tracking improve decision quality?" may generate experiments for different hypothesis types, different market conditions, and different confidence thresholds.

### 5.3 Layer 2 — Experiment

An experiment is a specific test design within a research proposal.

**Components:**
- **Experiment ID:** Unique identifier.
- **Research Proposal ID:** Link to the parent proposal.
- **Specific Hypothesis:** The precise claim being tested in this experiment.
- **Design:** How the test will be conducted. What data, what conditions, what controls.
- **Baseline:** What the result will be compared against.
- **Metrics:** Which metrics will be measured.
- **Analysis Method:** How the results will be evaluated.
- **Preregistration:** The design is documented before the experiment begins.

### 5.4 Layer 3 — Operational Decisions

Every operational decision — every trade, every monitoring action, every explicit non-action — is part of one or more experiments. Each decision produces a scientific record (Part IV) that contributes evidence to the experiment.

The same operational decision may contribute to multiple experiments. A single trade may simultaneously test a hypothesis about gold price movement and contribute to a broader experiment about hypothesis lifecycle effectiveness.

### 5.5 How the Layers Interact

1. A **research proposal** identifies a gap in criterion and frames a question.
2. One or more **experiments** are designed to test hypotheses derived from the proposal.
3. **Operational decisions** execute the experiments, producing scientific records.
4. **Experiment results** are compiled from the scientific records of all decisions in the experiment.
5. **Knowledge** is extracted from confirmed or rejected hypotheses.
6. **Criterion** is updated based on accumulated knowledge across experiments.

This hierarchy ensures that every operational decision has scientific purpose. No trade is an isolated event. Every trade belongs to an experiment. Every experiment serves a research question. Every research question addresses a gap in criterion.

## PART VI: The Three Memories

### 6.1 Three Independent Memory Systems

The redefined MVP maintains three independent memory systems. Each stores a different type of knowledge and serves a different function.

```
┌─────────────────────────────────────────────────────────────┐
│                    SCIENTIFIC MEMORY                         │
│                                                              │
│  Stores: Research proposals, experiments, scientific         │
│  records, results, analyses, validated and rejected          │
│  hypotheses, knowledge entries.                              │
│                                                              │
│  Purpose: Preserves the complete scientific history of        │
│  the project. Enables meta-analysis, reproducibility,        │
│  and longitudinal trend evaluation.                          │
│                                                              │
│  Mutability: Append-only. Scientific records are never       │
│  modified or deleted.                                        │
├─────────────────────────────────────────────────────────────┤
│                   OPERATIONAL MEMORY                         │
│                                                              │
│  Stores: Decisions, trades, actions, outcomes, failure       │
│  records, execution details, timing data.                    │
│                                                              │
│  Purpose: Enables real-time operational decisions and         │
│  short-term feedback loops. Supports the immediate           │
│  execution of the operational cycle.                         │
│                                                              │
│  Mutability: Append-only for closed cycles. Active cycles    │
│  may be updated until they are closed.                       │
├─────────────────────────────────────────────────────────────┤
│                    SEMANTIC MEMORY                            │
│                                                              │
│  Stores: Knowledge, patterns, criterion state, calibration   │
│  parameters, source reliability scores, priors, context       │
│  profiles, operational rules.                                │
│                                                              │
│  Purpose: Encodes what the system has learned. This is the   │
│  memory that directly informs future decisions.              │
│                                                              │
│  Mutability: Continuously updated as new knowledge is        │
│  extracted and new evidence refines existing knowledge.      │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 How They Interact

**Scientific Memory → Operational Memory:**
When a research proposal and experiment are defined, the experiment's requirements inform what operational data must be collected. Operational Memory extends its data collection to satisfy the experiment's needs.

**Operational Memory → Scientific Memory:**
When an operational cycle completes, its scientific record is written to Scientific Memory. The raw operational data (trades, outcomes) is linked to the scientific record but stored separately.

**Scientific Memory → Semantic Memory:**
When an experiment concludes, the knowledge extracted is written to Semantic Memory. This knowledge updates the system's criterion — its calibration, priors, patterns, and rules.

**Semantic Memory → Operational Memory:**
When an operational cycle begins, Semantic Memory provides the priors and patterns that inform hypothesis formation and evidence evaluation. The system's current criterion state shapes how new information is interpreted.

**Operational Memory → Semantic Memory:**
Short-term feedback loops (e.g., confidence calibration adjustment) update Semantic Memory directly without going through the full scientific cycle. These are quick refinements based on recent operational outcomes.

### 6.3 Why Three Memories Instead of One

Three memories exist because they have different purposes, different mutability requirements, and different query patterns:

- **Scientific Memory** is immutable and historical. It answers: "What did we know and when did we know it?"
- **Operational Memory** is current and actionable. It answers: "What is happening now and what just happened?"
- **Semantic Memory** is integrated and judgmental. It answers: "What do we believe and how confident are we?"

A single memory would force tradeoffs between immutability (required for scientific integrity) and updatability (required for operational responsiveness). Three memories with explicit interaction rules preserve both.

## PART VII: The MVP as a Laboratory

### 7.1 Every Trade Is an Experiment

In the redefined MVP, there are no trades. There are only experiments that involve capital allocation.

A trade without a hypothesis is not a trade. It is noise. It consumes capital, produces a result, and teaches nothing.

Every trade must be linked to:
- The hypothesis it tests
- The experiment it belongs to
- The research proposal it serves
- The evidence that supported it
- The confidence at decision time
- The predicted scenario
- The invalidation conditions
- The outcome relative to prediction
- The error classification
- The knowledge extracted

A trade that cannot answer these questions is not an experiment. It is a gamble.

### 7.2 Every Hypothesis Is Temporary

No hypothesis survives forever. A confirmed hypothesis remains provisional — it may be falsified by future evidence. A rejected hypothesis is preserved for meta-analysis but is no longer acted upon.

The half-life of a hypothesis depends on:
- The rate of change in its domain (faster change = shorter half-life)
- The strength of supporting evidence (stronger evidence = longer half-life)
- The specificity of its predictions (more specific = more easily falsified = shorter half-life)
- The breadth of conditions it has survived (more conditions = longer half-life)

### 7.3 Every Failure Is Valuable

A failure is not a loss. It is an experiment that produced clear evidence against a hypothesis.

The value of a failure depends on:
- **Clarity:** Does the failure clearly reject the hypothesis, or is it ambiguous?
- **Classifiability:** Can the failure be classified into a specific error type?
- **Generalizability:** Does the failure teach a lesson that applies beyond this specific case?
- **Novelty:** Has this type of failure been seen before, or is it a new discovery?

A failure that clearly rejects a hypothesis and teaches a generalizable lesson is more valuable than a success that confirms a trivial hypothesis.

### 7.4 Every Decision Produces Evidence

Even decisions not to act produce evidence. The system's decision to ignore an opportunity, to wait for more evidence, or to reduce a position — each of these is a decision that records reasoning, expectation, and outcome.

The scientific record captures all decisions, not only executed trades. This prevents survivorship bias — the system cannot selectively remember only the decisions that led to actions.

### 7.5 Every Experiment Updates Criterion

The purpose of every experiment is to update the system's criterion. An experiment that does not change the system's judgment is an experiment that taught nothing.

The criterion update may be small — a slight adjustment to confidence calibration, a minor revision to evidence source reliability. Or it may be large — a new pattern recognized, a previous knowledge entry invalidated, a new research question generated. But some update must occur.

If the system completes an operational cycle and its criterion is unchanged, the cycle failed — not because the trade may have lost money, but because no learning occurred.

### 7.6 The MVP Exists to Learn

The redefined MVP is not a production system. It is not evaluated by uptime, throughput, or profitability. It is evaluated by:
- How many hypotheses were tested?
- How much knowledge was extracted?
- Did criterion metrics improve?
- What was learned that was not known before?

A day in which the MVP generates no knowledge is a day the MVP failed — regardless of profits.

A day in which the MVP generates significant knowledge but loses money is a day the MVP succeeded — the loss is an investment in better future judgment.

## PART VIII: Success Metrics

### 8.1 Three Independent Dimensions

The redefined MVP must be evaluated across three independent dimensions. No single dimension is sufficient.

| Dimension | Measures | Primary Audience | Risk if Over-optimized |
|-----------|----------|-----------------|------------------------|
| **Trading Success** | Is the system profitable and well-behaved? | Operators, stakeholders | Optimizing for profit without learning |
| **Criterion Success** | Is the system developing better judgment? | Scientists, researchers | Optimizing for metrics without operational value |
| **Research Success** | Is the system producing valid knowledge? | The research program | Optimizing for publications without criterion improvement |

### 8.2 Trading Success Metrics

These metrics measure the operational health of the laboratory. They are necessary but not sufficient.

| Metric | What It Measures | Limitation |
|--------|-----------------|------------|
| Profit Factor | Gross profit / gross loss | Does not measure learning |
| Win Rate | Proportion of profitable trades | Does not measure risk or magnitude |
| Maximum Drawdown | Largest peak-to-trough decline | Does not measure recovery or learning |
| Sharpe Ratio | Risk-adjusted return | Can be gamed by reducing position sizing |
| Exposure Time | Proportion of time capital is deployed | Does not measure decision quality |
| Capital Efficiency | Return per unit of capital deployed | Does not measure criterion improvement |
| Opportunity Capture Rate | Profitable opportunities acted upon / total that existed | Extremely difficult to measure |

Trading success metrics answer: "Is the laboratory operating without destroying itself?"

### 8.3 Criterion Success Metrics

These metrics measure whether the system's judgment is improving. They are the primary evidence for the project's central hypothesis.

| Metric | What It Measures | Limitation |
|--------|-----------------|------------|
| Reasoning Quality | Coherence, rigor, and completeness of reasoning before action | Scoring may be subjective |
| Question Quality | Specificity, relevance, and testability of questions asked | Hard to automate measurement |
| Hypothesis Quality | Precision, falsifiability, and evidence grounding of hypotheses | Requires sufficient sample size |
| Calibration | Alignment between confidence and accuracy | Requires multiple confidence buckets with sufficient data |
| Learning Rate | Speed and consistency of knowledge extraction | Quantity ≠ quality of lessons |
| Error Recurrence Rate | Proportion of hypotheses whose error type matches previous errors | Requires sufficient error classification |
| Evidence Quality | Relevance, independence, and source reliability of evidence used | Requires ground truth for evaluation |
| Uncertainty Reduction | How quickly confidence converges as evidence accumulates | Requires comparable scenarios |
| Decision Compression | Ability to make good decisions with less evidence over time | Requires baseline for comparison |
| Adaptability | How quickly the system adjusts to regime changes | Requires regime change events |

Criterion success metrics answer: "Is the system developing the ability to judge what matters?"

### 8.4 Research Success Metrics

These metrics measure whether the scientific process is producing valid, reusable knowledge.

| Metric | What It Measures | Limitation |
|--------|-----------------|------------|
| Hypotheses Tested | Number of hypotheses that completed a full lifecycle | Quantity ≠ quality |
| Hypotheses Confirmed | Proportion of tested hypotheses that were confirmed | Confirmation may be due to weak tests |
| Hypotheses Rejected | Proportion of tested hypotheses that were rejected | High rejection rate may indicate poor hypothesis formation |
| Knowledge Produced | Number of generalizable lessons extracted | Lessons may be trivial |
| Knowledge Applicability | Proportion of knowledge entries that informed future decisions | Requires tracking knowledge use |
| Experiment Reproducibility | Proportion of experiments that produced consistent results when repeated | Requires repeated experiments |
| Evidence Quality Trend | Is evidence quality improving across cycles? | Requires longitudinal analysis |
| Research Velocity | Time from question to answer across the full cycle | Speed ≠ quality |

Research success metrics answer: "Is the scientific process producing valid knowledge?"

### 8.5 Why All Three Dimensions Are Necessary

- **Trading success without criterion success** means the system is profitable but not learning. It will be overtaken when conditions change.
- **Criterion success without research success** means the system appears to improve but the improvement cannot be attributed to the scientific process. It may be coincidence, market conditions, or unmeasured confounds.
- **Research success without trading success** means the system is producing knowledge that does not translate to operational decisions. The knowledge may be irrelevant or wrong.
- **Trading success + research success without criterion success** means the system is profitable and running good experiments, but the experiments are not improving judgment. The research program is failing its core mission.
- **All three improving** is the only condition that justifies the claim that the system is developing operational criterion.

## PART IX: Exit Criteria

### 9.1 When Is the MVP Complete?

The MVP is complete when it consistently produces scientific evidence that the system is developing criterion.

Not when it is profitable. Profit is a consequence, not a completion criterion.

Not when it trades autonomously. Autonomy is a capability, not a milestone.

Not when it achieves a certain Sharpe ratio or win rate. These are trading metrics, not criterion metrics.

### 9.2 Measurable Completion Criteria

The MVP is complete when all of the following conditions are met:

**1. Scientific record framework is operational.**
Every operational cycle produces a complete scientific record (Part IV) with all required fields populated. No trade occurs without a linked hypothesis. No outcome is recorded without reflection. No cycle completes without knowledge extraction.

**2. Baseline criterion metrics are established.**
At least 100 scientific records with complete data exist. Baseline measurements for calibration, hypothesis quality, evidence quality, error recurrence rate, and knowledge yield are documented. The baseline has sufficient sample size for statistical comparison.

**3. Criterion improvement is detectable.**
After the baseline is established, a statistically significant positive trend (p < 0.10) is detectable in at least one criterion success metric over a 90-day rolling window with a minimum of 30 decisions per window.

**4. The improvement is attributable.**
The improvement can be traced to specific changes in the system's reasoning — better hypotheses, better evidence use, better calibration — and is not explainable by external factors (market regime, data volume, human intervention).

**5. The research protocol is followed.**
At least 80% of operational cycles demonstrate compliance with the Research Protocol (18_RESEARCH_PROTOCOL.md): hypotheses are falsifiable, experiments have baselines, failures are classified, knowledge is extracted.

**6. Evidence decays.**
The system demonstrates that older evidence is appropriately discounted. Knowledge from six months ago receives less weight than knowledge from yesterday, unless explicitly justified.

**7. Rejected hypotheses are preserved.**
Every rejected hypothesis is archived with complete lifecycle data. The archive is searchable and queryable. Meta-analysis of rejected hypotheses produces actionable insights.

### 9.3 MVP Completion Is Not Project Completion

Meeting the exit criteria means the laboratory is operational. It does not mean the project is finished. It means the project has built a working scientific instrument that can now begin its real work: producing evidence of criterion development over years and across domains.

After MVP completion:
- The system continues to operate and accumulate evidence
- Criterion metrics are tracked and reported continuously
- New research proposals are generated based on gaps discovered during operations
- The experimental apparatus is refined based on operational experience
- New domains are added when criterion evidence warrants expansion

## PART X: The Long-Term Vision

### 10.1 The Invariant Core

Throughout the project's evolution, one thing remains invariant: criterion.

Criterion is the accumulated ability to judge what matters. It is the output of the learning process, the measure of the system's maturity, the purpose of every experiment, and the evidence that the project is succeeding.

Domains change. Experiments change. Hypotheses change. Architectures change. Criterion endures.

### 10.2 The Domain Trajectory

The project envisions expanding through domains, each building on the criterion developed in the previous domain.

**Phase 1 — Trading (Current):**
- Fast feedback (hours to days)
- Unambiguous outcomes (profit/loss)
- Abundant data (thousands of cycles per year)
- Purpose: Develop and validate the criterion framework in the most favorable experimental environment.

**Phase 2 — Creator:**
- Medium feedback (days to weeks)
- Partially subjective outcomes (engagement, reach, quality scores)
- Moderate data (hundreds of cycles per year)
- Purpose: Test whether criterion developed in trading transfers to a domain with slower feedback and more subjective outcomes.
- Trigger: The system demonstrates consistent criterion improvement in trading across multiple market regimes (minimum 2 years of data).

**Phase 3 — Entrepreneur:**
- Slow feedback (weeks to months)
- Complex outcomes (revenue, team, market position)
- Scarce data (tens of cycles per year)
- Purpose: Test whether criterion operates in domains with long feedback loops, where each decision has high stakes and few repetitions.
- Trigger: The system demonstrates criterion transfer from trading to creation (positive criterion metrics in both domains).

**Phase 4 — General Decision Intelligence:**
- Variable feedback depending on domain
- All outcome types
- All data volumes
- Purpose: Demonstrate that the criterion framework is domain-independent — that it improves decision quality in any domain where the system operates.
- Trigger: Demonstrated criterion improvement in at least three fundamentally different domains.

### 10.3 Criterion Remains the Core

At every phase, the question is the same: "Did the system improve its criterion?"

The answer is always provisional. The trajectory is never guaranteed. The work is never finished.

### 10.4 The Ten-Year View

In year 10:
- The system has accumulated thousands of scientific records across at least three domains
- Criterion metrics show consistent improvement over the entire decade
- The system operates at Autonomy Level 5 or higher
- The project has contributed validated methodology to the scientific community
- The documents and protocols have been revised multiple times based on evidence
- The project has survived multiple regime changes, technology shifts, and domain transitions
- The core question remains the same: "Did the system improve its criterion?"

## Self-Review

### S-01: Weak Assumptions

**Assumption: Scientific records can be generated automatically.**
The schema in Part IV contains 30+ fields. Some fields (reflection, knowledge extraction, error classification) require judgment that may not be automatable with current technology. The MVP may need human-in-the-loop for these fields initially.

**Assumption: Three memories are necessary.**
The three-memory architecture (Scientific, Operational, Semantic) adds complexity. A simpler design might use two memories (immutable record + updatable state) or even one memory with versioned records. The three-memory design should be tested against a simpler alternative.

**Assumption: Criterion metrics converge with trading metrics over time.**
The project assumes that improving criterion eventually produces improving profits. This is plausible but unproven. If criterion improves but profits do not follow, either criterion is the wrong goal or the metrics are measuring the wrong thing.

**Assumption: The operational cycle (Part III) is complete.**
The 16-stage cycle may omit stages that matter. For example, there is no explicit "generate alternatives" stage for the decision (only for hypotheses). The cycle may need a "decision alternatives" stage between confidence and decision.

### S-02: Architectural Risks

**Record volume may overwhelm analysis.**
If the system generates hundreds of scientific records per year, manual analysis becomes infeasible. Automated analysis tools must be developed in parallel with the record system.

**The three memories may become inconsistent.**
Without explicit synchronization rules, the three memories may diverge. Operational Memory may record an outcome that Scientific Memory never received. Semantic Memory may be updated based on knowledge that Operational Memory has not yet applied.

**Research proposal overhead may slow operations.**
If every trade requires a research proposal, experiment design, and scientific record, the operational tempo may drop to levels that generate insufficient data. The system needs a lightweight path for exploratory trades that do not require full research proposals.

### S-03: Concepts Lacking Evidence

**The three-layer research hierarchy (Proposal → Experiment → Decision).**
This hierarchy is designed but untested. It may prove too rigid for practical use. The relationship between layers may need to be more fluid.

**The 30+ field scientific record.**
The complete record is comprehensive but may be impractical. Some fields may prove redundant. Others may prove insufficient. The record schema should be iterated based on operational experience.

**The domain trajectory (Trading → Creator → Entrepreneur → General).**
This trajectory assumes criterion transfers between domains. Assumption A-006 rates this as Low-Medium confidence. The entire long-term vision rests on an assumption that the project itself identifies as its weakest.

### S-04: What Should Be Challenged

**Is the 16-stage cycle too detailed?**
The original MVP had a much simpler cycle: events → agents → council → trade. The redesigned cycle has 16 stages. Each stage is justified, but the aggregate complexity may slow the system to the point where it cannot generate enough cycles to develop criterion. A simpler cycle that captures the essential stages (hypothesis → action → outcome → reflection) may be more practical.

**Are three success dimensions too many?**
Trading, criterion, and research success are all tracked. But if the project must report on all three, there is a risk of metric overload. The project may lose focus on what matters — criterion improvement — by trying to optimize three dimensions simultaneously. Consider whether research success and criterion success should be merged.

**Is the laboratory metaphor fully appropriate?**
A laboratory implies controlled conditions and deliberate experiments. Trading markets are not controlled environments. The system cannot run A/B tests in the same market at the same time. The laboratory metaphor is useful but has limits — the project cannot control its experimental conditions.

### S-05: Proposed Simplifications

1. **Reduce the operational cycle to 10 essential stages** if the 16-stage version proves too slow: Need → Question → Hypothesis → Evidence → Decision → Action → Outcome → Reflection → Knowledge → Criterion Update.

2. **Merge Scientific Memory and Operational Memory** into a single append-only store with different indexing for scientific and operational queries. This eliminates the synchronization problem.

3. **Define a minimal scientific record** (15 fields) for exploratory cycles and a full scientific record (30 fields) for confirmatory cycles. This provides a fast path for low-risk experiments.

4. **Combine criterion success and research success** into a single dimension: "learning success." This reduces metric overload and focuses on the core question: is the system learning?

### S-06: Open Questions for Future Iterations

1. How should research proposals be prioritized when multiple gaps in criterion exist simultaneously?
2. Who or what reviews the quality of scientific records and ensures protocol compliance?
3. How should the system handle experiments that produce no clear outcome (inconclusive)?
4. What is the minimum number of cycles needed to detect a statistically significant trend in criterion metrics?
5. Should the system ever run experiments that are expected to reduce short-term profits in order to test a high-potential hypothesis?
