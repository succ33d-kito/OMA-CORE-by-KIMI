# Reasoning Validation

*Version 1.0 — June 2026*
*Learning Core — Layer 5*

---

## 1. Purpose

Reasoning modes must be evaluated, not trusted. Every reasoning mode is a hypothesis about how to think — it may be correct in some conditions and incorrect in others. The system must track reasoning mode performance and adjust its use of each mode based on evidence.

This document defines the metrics, feedback loops, retirement criteria, and safeguards for reasoning mode validation.

---

## 2. Core Principle

A reasoning mode's effectiveness is measured by the quality of hypotheses it produces, not by the elegance of its reasoning.

---

## 3. Validation Metrics

### Primary Metrics

| # | Metric | Definition | Why It Matters | Minimum Sample |
|---|--------|------------|----------------|----------------|
| 1 | **Hypothesis confirmation rate** | % of hypotheses generated by this mode that receive CONFIRMED verdict | Direct measure of hypothesis quality | 20 hypotheses |
| 2 | **False hypothesis rate** | % of hypotheses generated by this mode that receive REJECTED verdict | Measures how often the mode leads the system astray | 20 hypotheses |
| 3 | **Discard rate** | % of scenarios from this mode that are discarded by the Reasoning Council (below threshold) | Measures how often the mode produces low-quality output | 50 scenarios |
| 4 | **Average Council score** | Mean evaluation score across all scenarios from this mode | Measures raw output quality before filtering | 50 scenarios |
| 5 | **Time-to-validation** | Average time from hypothesis creation to outcome comparison | Measures how quickly the mode's predictions can be tested | 20 hypotheses |

### Secondary Metrics

| # | Metric | Definition | Why It Matters |
|---|--------|------------|----------------|
| 6 | **Novelty usefulness** | % of novel hypotheses (novelty score >0.7) from this mode that are confirmed | Measures whether the mode produces genuinely new and correct insights |
| 7 | **Duplicate rate** | % of scenarios from this mode that are discarded as duplicates | Measures how often the mode rehashes existing ideas |
| 8 | **Knowledge yield** | Knowledge items generated from hypotheses produced by this mode, per hypothesis | Measures whether the mode produces learning even when wrong |
| 9 | **Opportunity cost impact** | Average opportunity cost of hypotheses generated by this mode that were REJECTED | Measures the cost of following this mode's wrong predictions |
| 10 | **Error type distribution** | Distribution of error types for rejected hypotheses from this mode | Reveals what kind of mistakes each mode tends to make |

### Calibration Metrics

| # | Metric | Definition | Why It Matters |
|---|--------|------------|----------------|
| 11 | **Confidence calibration** | Does the mode's self-reported confidence match actual confirmation rate? | An overconfident mode should be discounted; an underconfident mode should be amplified |
| 12 | **Mode Brier score** | Mean squared error between mode confidence and outcome | Standard calibration metric |
| 13 | **Condition-specific calibration** | Calibration broken down by market condition (trending, ranging, volatile) | Reveals which conditions the mode handles well or poorly |

## 5. How Reasoning Performance Feeds Knowledge

When a reasoning mode produces a hypothesis that is later confirmed or rejected, the system learns about the mode itself:

| Outcome | Knowledge Generated | Example |
|---------|---------------------|---------|
| Mode consistently confirmed | "Mode X is reliable under conditions Y" | "Causal reasoning in trending markets has 72% confirmation rate. Prioritize causal hypotheses during trends." |
| Mode consistently rejected | "Mode X is unreliable under conditions Y" | "Risk reasoning during low-volatility regimes produces 70% false positives. Discount risk scenarios in low vol." |
| Mode calibration improving | "Mode X is learning — confidence is converging with accuracy" | "Temporal reasoning calibration improved from 0.31 to 0.22 Brier over 90 days. Mode is learning." |
| Mode calibration diverging | "Mode X is becoming overconfident — investigate" | "Economic reasoning confidence has increased to 0.80 but confirmation rate is 0.35. Mode is overconfident." |

This knowledge feeds into the Knowledge Object lifecycle (docs/learning_core/02_KNOWLEDGE_OBJECT.md) and eventually into Criterion Evolution (docs/learning_core/03_CRITERION_EVOLUTION.md).

## 6. How Reasoning Performance Affects Criterion

Reasoning mode performance directly impacts Criterion dimensions:

| Criterion Dimension | How Mode Performance Affects It |
|---------------------|----------------------------------|
| **Hypothesis Quality** | Modes with high confirmation rates should produce more hypotheses; modes with low rates should be limited |
| **Decision Quality** | If modes that produce confirmed hypotheses are ignored by the Agent Council, decision quality suffers. Alignment between reasoning and action is a Criterion signal. |
| **Calibration** | Mode calibration aggregated across all modes is a component of system calibration |
| **Error Recurrence** | If a mode consistently produces WRONG_TIMING errors, the system learns to add temporal reasoning to that mode's output |
| **Learning Velocity** | Modes that improve their confirmation rate over time demonstrate learning velocity at the reasoning level |
| **Knowledge Yield** | Modes that produce high knowledge yield even when wrong are valuable — they generate learning from failure |

## 7. Mode Weight Adjustment

The Reasoning Council weights modes by their historical performance:

### Weight Formula (V1)

```
mode_weight = base_weight × confirmation_rate_adjustment × calibration_adjustment

base_weight = initial weight (equal across all V1 modes = 0.20 for 5 modes)

confirmation_rate_adjustment = max(0.5, min(2.0, mode_confirmation_rate / average_confirmation_rate))

calibration_adjustment = max(0.5, min(1.5, 1.0 - (mode_brier - average_brier)))
```

### Weight Adjustment Rules

| Condition | Action |
|-----------|--------|
| Mode confirmation rate > average + 20% over 30+ hypotheses | Increase weight by 0.05 per review cycle |
| Mode confirmation rate < average - 20% over 30+ hypotheses | Decrease weight by 0.05 per review cycle |
| Mode Brier score > 0.3 (poor calibration) | Flag for review. Cannot increase weight until calibration improves. |
| Mode Brier score < 0.15 (excellent calibration) | Bonus weight increase of 0.02 |
| Mode produces no hypotheses for 30 days | Suspend. Re-evaluate after 7 days. |

### Minimum Thresholds Before Weight Change

| Weight Change | Minimum Hypotheses | Minimum Time |
|--------------|-------------------|--------------|
| Increase | 30 | 30 days |
| Decrease | 20 | 14 days |
| Suspend | 50 with <20% confirmation | 60 days |

## 8. Mode Retirement

A mode may be retired when it consistently underperforms:

| Retirement Condition | Criteria | Action |
|---------------------|----------|--------|
| **Performance floor** | Confirmation rate < 20% over 50+ hypotheses across 2+ market regimes | Status → RETIRED. Move to reasoning mode graveyard. |
| **Zero novel output** | 90%+ of scenarios discarded as duplicates over 60 days | Status → RETIRED. Mode is not contributing new ideas. |
| **Negative knowledge yield** | Knowledge yield < 0.05 per hypothesis over 50+ hypotheses | Status → UNDER_REVIEW. If no improvement after 30 days, RETIRED. |
| **Resource cost exceeds value** | Mode consumes more compute/time than its hypotheses are worth | Status → UNDER_REVIEW. Calculate cost/benefit ratio. |
| **Superseded** | A new mode covers the same domain with better performance | Status → SUPERSEDED. Reference to the replacement mode. |

### Mode Graveyard

Retired modes are recorded in a reasoning mode graveyard:

```
{
  "mode_id": "analogical",
  "status": "RETIRED",
  "active_period": "2026-08-01 to 2027-01-15",
  "total_hypotheses": 87,
  "confirmation_rate": 0.18,
  "retirement_reason": "Confirmation rate below 20% threshold across 87 hypotheses over 2 regimes. Mode produced no novel insights.",
  "lessons_learned": "Analogical reasoning failed because market conditions were structurally different from historical analogs. Similarity matching was surface-level.",
  "revival_condition": "If analogical matching is redesigned to use structural (not surface) similarity, mode may be re-implemented.",
  "retired_at": "2027-01-15"
}
```

## 9. Minimum Evidence Thresholds

Before any claim about reasoning mode performance is accepted:

| Claim | Minimum Evidence |
|-------|------------------|
| "Mode X has H% confirmation rate" | 20+ hypotheses from mode X, with outcome comparisons |
| "Mode X is better than mode Y" | 30+ hypotheses from each, in overlapping conditions |
| "Mode X is reliable in condition C" | 15+ hypotheses from mode X in condition C |
| "Mode X should be retired" | 50+ hypotheses, <20% confirmation, 2+ regimes |
| "Mode X calibration is poor" | 30+ hypotheses with confidence and outcome records |

## 10. Danger of Overfitting Reasoning Modes

| Risk | Description | Mitigation |
|------|-------------|------------|
| **Mode weight herding** | Modes that happened to be correct in recent conditions gain all the weight, reducing diversity | Weight change limits per cycle. Minimum weight floor (0.10) for all active modes. |
| **Regime-specific over-optimization** | Modes are weighted for the current regime, leaving the system vulnerable to regime change | Track mode performance by regime. If regime changes, reset weights to baseline. |
| **Confirmation bias** | The system favors modes that confirm existing beliefs, avoiding contradictory evidence | Track whether the system is generating enough competing hypotheses. If competing hypothesis rate drops below 10%, investigate. |
| **Mode extinction** | A temporarily underperforming mode gets retired before it can recover | Long retirement thresholds (50+ hypotheses, 2+ regimes). Revival condition always documented. |

## 11. Periodic Review

| Rhythm | Review Activity | Output |
|--------|----------------|--------|
| **Weekly** | Check mode performance metrics. Flag any mode with sudden performance changes. | Updated mode dashboard |
| **Monthly** | Review mode weights. Adjust if evidence supports. Review any mode UNDER_REVIEW. | Weight adjustment decisions |
| **Quarterly** | Review all mode definitions. Are new modes needed? Should any be merged? Are the mode boundaries still correct? | Updated mode registry |
| **Yearly** | Full reasoning framework audit. Are the reasoning modes producing better hypotheses than last year? Is the framework itself still valid? | Reasoning framework health report |
